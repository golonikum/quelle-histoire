# React Virtual DOM & Reconciliation: Как React обновляет UI

React Virtual DOM — это **легковесная JS-структура**, представляющая реальный DOM в памяти. Reconciliation (diffing) — алгоритм минимизации изменений при обновлении UI. React сравнивает **новый VDOM со старым** и применяет **только разницу**.

## Как работает (шаг за шагом)

```
1. JSX → React.createElement → VDOM объект
2. setState → React ререндерит компонент → новый VDOM
3. Diffing: новый VDOM vs предыдущий → патч
4. Патч применяется к реальному DOM
```

## Наглядное представление VDOM

```
Реальный DOM:
<html>
  <div class="container">
    <h1>Count: 1</h1>
  </div>
</html>

Virtual DOM (JS объект):
{
  type: 'html',
  props: null,
  children: [{
    type: 'div',
    props: { className: 'container' },
    children: [{
      type: 'h1',
      props: null,
      children: ['Count: 1']
    }]
  }]
}
```

## Пример diffing

**До** (`count = 1`):

```
VDOM1 = {
  type: 'div',
  children: ['Count: 1', 'Count: 2']
}
```

**После** (`count = 5`):

```
VDOM2 = {
  type: 'div',
  children: ['Count: 5', 'Count: 2']
}
```

**Diff результат**: заменить `children[^0]` с `'Count: 1'` на `'Count: 5'`

```
DOM change: document.querySelector('div').children[^0].textContent = 'Count: 5'
```

## Алгоритм Reconciliation (O(n) эвристика)

```
1. Разные типы → полная перезапись
   <div> → <span> → пересоздать всё

2. Тот же тип → сравнить props → обновить атрибуты
   className: 'old' → 'new' → setAttribute()

3. Дети → рекурсивно по индексам + keys
   key="id1" помогает идентифицировать перемещения
```

## Визуализация процесса

```
ReactDOM.render(<App count={1} />)
  ↓ JSX
React.createElement('div', {className: 'app'}, 'Count: 1')
  ↓ VDOM объект
Fiber tree (внутренняя структура React)
  ↓ setState(count=2)
Новый VDOM → diff → минимальные DOM мутации
```

## ASCII Fiber дерево

```
App (fiber)
├── div.container (fiber)
│   └── h1#title (fiber) [text: "Count: 5"]
└── button (fiber) [onClick handler]
```

## Почему эффективно?

| Подход              | Изменения         | DOM операций |
| :------------------ | :---------------- | :----------- |
| `innerHTML = '...'` | Полная перезапись | 100+         |
| **React VDOM**      | Только текст      | **1**        |
| jQuery `.html()`    | Полная перезапись | 100+         |

## React 18+ Fiber

**Современная реализация**: прерывистый рендер, приоритет задач:

```
Render phase (создаёт VDOM, можно прервать)
↓
Commit phase (применяет к DOM, атомарно)
```

**Код React** (упрощённо):

```jsx
function updateContainer(element) {
  const newTree = getVDOM(element); // JSX → дерево объектов
  const diff = diffTrees(prevTree, newTree); // Рекурсивный diff
  applyPatches(realDOM, diff); // Минимальные мутации
}
```

**Профайлер**: React DevTools → "Profiler" → смотрите ререндеры и diff.

# React Fiber

`React Fiber` — это **новый алгоритм React** (с версии 16), который делает рендер **прерываемым**. Представь, что раньше React рендерил всё сразу и если список из 10 000 элементов — браузер зависал на 200мс. Теперь React может **начать рендер → сделать одну строчку → пауза → клик пользователя → продолжить**.

`React Fiber` — это фундамент, на котором строятся `Concurrent Mode`, `Suspense` и `useTransition`. Вместе они решают главную проблему: тяжёлые обновления не должны блокировать UI.

## Простая аналогия: игра Tetris

**До Fiber** (старый React):

```
Падают 10 фигур сразу → UI зависает → игрок не успевает повернуть → Game Over
```

**Fiber** (новый React):

```
Падает 1 фигура → игрок поворачивает → падает 2-я → игрок кликает → и т.д.
```

## Как это работает?

### 1. React разбивает работу на **кусочки** (Fiber Nodes)

```
Компонент App состоит из:
✅ Пункт 1: <Header />
✅ Пункт 2: <List items={10000} />
⏸️  Пункт 3: <Footer /> ← Пауза, браузер свободен
```

### 2. **Два дерева** вместо одного

```
Current Tree (то что на экране)
App ── Header ── Logo

Work In Progress (строится)
App ── Header ── Logo ✓
     │
     └─ List (1000/10000) ⏳ ← React может прервать здесь
```

### 3. **Планировщик** решает приоритеты

```
🔴 Высокий: клик пользователя, анимация
🟡 Средний: загрузка данных
🟢 Низкий: автосохранение черновика
```

## Зачем тебе как мидлу это знать?

### 1. **Понимаешь StrictMode**

```jsx
// Почему дважды рендерится в dev-режиме?
function App() {
  console.log("render"); // 1️⃣ 2️⃣
}
```

**Ответ**: React Fiber тестирует, что твой компонент **устойчив к прерыванию**.

### 2. **useTransition** (React 18)

```jsx
const [start, setStart] = useTransition();
<button onClick={() => {
  setStart(true);
  setTimeout(() => {
    setFilter('complex'); // Низкий приоритет
    setStart(false);
  }, 0);
}}>
```

**Фича Fiber**: React сначала покажет кнопку "Загрузка...", потом тяжёлый фильтр.

### 3. **Профайлер показывает Fiber**

React DevTools → Profiler → видишь **какие именно Fiber-ноды** рендерятся.

## Наглядная схема

```
👆 Пользователь кликает
   ↓
📋 React ставит задачу в очередь (по приоритету)
   ↓
⚙️  Планировщик: "У меня 16мс до следующего кадра"
   ↓
🔨 Fiber: "Сделаю Header → List (100 элементов) → Пауза"
   ↓
🎬 Браузер: рисует кадр 60fps
   ↓
🔄 Повтор (если время осталось)
```

## Простая проверка в коде

```jsx
function SlowList() {
  console.log("🔨 Рендерю 1000 элементов...");
  const items = Array(1000)
    .fill()
    .map((_, i) => <div key={i}>Item {i}</div>);
  return <div>{items}</div>;
}
```

**В StrictMode увидишь**: рендер → пауза → повтор → commit.

## Итог для мидла

**Fiber = React стал умнее**. Вместо "сделать всё или ничего" — "делай понемногу, чтобы UI не подвисал". Это основа `useTransition`, `Suspense`, Concurrent Mode.

**Не зубри внутренности** — понимай принцип **прерываемости** и пользуйся новыми API.
