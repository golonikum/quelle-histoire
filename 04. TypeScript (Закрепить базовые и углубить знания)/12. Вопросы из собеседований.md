# Старые

## Зачем писать `const enum`? Что это дает?

`const enum` — это оптимизированная версия enum, которая полностью исчезает при компиляции в JavaScript. Значения встраиваются inline.

### 🚨 **Ограничения `const enum`**

#### ❌ **НЕ работает с Object.keys()**

```ts
const enum Status {
  Success = 200,
}
const obj = { status: Status.Success }; // obj.status = 200

Object.keys(obj); // ❌ Ошибка! TS не знает про enum
```

#### ❌ **НЕ работает в .d.ts**

```ts
// ❌ В .d.ts файлах запрещено!
const enum Status {
  OK,
}
// Ошибка компиляции
```

#### ❌ **Только литералы**

```ts
const enum Status {
  OK = 200,
  // ERROR = someFunction()  ❌ Вычисляемые значения запрещены
}
```

**Ответ:**

1. ✅ Inline-замена (0 байт в бандле)
2. ✅ Быстрее выполнение
3. ✅ Tree-shaking работает
4. ✅ Типизация такая же
   ❌ Минус: нет Object.keys(), только литералы

const enum = **типобезопасность + 0 байт в продакшене!**

## Как описать функцию, которая возвращает расширенный параметр

```
function fn<TParam>(param: TParam): TParam & { name: string; } {
   ...
}
```

## Type predicate

```
function isFish(pet: Fish | Bird): pet is Fish {
  return (pet as Fish).swim !== undefined;
}
```

# ИИ

## Дан тип: `type IsString<T> = T extends string ? 'yes' : 'no'`. Какой будет результат типа `IsString<string | number>` при стандартной (распределённой) семантике условных типов?

**Ответ:**

Условный тип `IsString<T> = T extends string ? 'yes' : 'no'` при подстановке объединения `string | number` вычисляется как `'yes' | 'no'`. Это происходит из-за того, что условные типы в TypeScript имеют распределяющее поведение по объединениям, когда T находится в позициях `extends`. Выражение `IsString<string | number>` де-факто разворачивается в `IsString<string> | IsString<number>`, что и даёт объединение `'yes' | 'no'`. Это ключевой механизм, который используют для построения типов-фильтров, реализации `Exclude`, `Extract` и других утилит.
