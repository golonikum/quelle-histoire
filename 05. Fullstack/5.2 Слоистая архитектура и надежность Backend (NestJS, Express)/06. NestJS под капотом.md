## 🔍 **NestJS под капотом — Middle+ (с подробными объяснениями)**

### **🎛️ 1. Application Bootstrap — последовательность запуска**

**Что происходит при `NestFactory.create(AppModule)`:**

1. **Создание ApplicationContext** — главная "песочница" приложения, где живут все провайдеры, контроллеры и модули. Это как "главный IoC контейнер", который знает обо всех зависимостях.
2. **Рекурсивная загрузка модулей** — начиная с `AppModule`, NestJS проходит по графу зависимостей (`imports`), загружая все модули **сверху вниз**. Если `AppModule` импортирует `UsersModule`, а тот `AuthModule` — создается **DAG (Directed Acyclic Graph)**.
3. **Сканирование метаданных** — `MetadataScanner` читает **все декораторы** (`@Controller`, `@Injectable`, `@Get`) через `Reflect.metadata`. TypeScript должен включить `"emitDecoratorMetadata": true`.
4. **Регистрация HTTP роутов** — `RouterExplorer` находит все методы с `@Get/@Post` и регистрирует их в **Express/Fastify** адаптере.
5. **Инициализация singleton провайдеров** — все `@Injectable()` создаются **один раз** при старте (если `scope: DEFAULT`).

**Визуализация**:

```
NestFactory.create()
→ ApplicationContext (IoC)
→ AppModule → UsersModule → AuthModule (DAG)
→ Controllers: express.get('/users', handler)
→ Providers: new UsersService(prisma)
→ app.listen(3000)
```

### **🏗️ 2. IoC Container — сердце системы DI**

**Как работает инъекция зависимостей:**

**TypeScript генерирует метаданные**:

```typescript
class UsersService {
  constructor(private prisma: PrismaService) {}
}
// → Reflect.defineMetadata('design:paramtypes', [PrismaService], UsersService)
```

**NestJS Container делает**:

```typescript
// Упрощенная логика
class IoCContainer {
  private instances = new Map<Token, any>();

  get<T>(token: Token): T {
    if (this.instances.has(token)) return this.instances.get(token);

    // 1. Читает конструктор
    const paramTypes = Reflect.getMetadata("design:paramtypes", token);

    // 2. Рекурсивно создает зависимости
    const deps = paramTypes.map((dep) => this.get(dep));

    // 3. Создает экземпляр
    const instance = new token(...deps);
    this.instances.set(token, instance);

    return instance;
  }
}
```

**Scopes (жизненный цикл)**:

- **DEFAULT (singleton)** — 1 экземпляр на всё приложение
- **REQUEST** — новый экземпляр на каждый HTTP запрос (с `REQUEST` контекстом)
- **TRANSIENT** — новый экземпляр при каждой инъекции

### **🧩 3. Module System — граф зависимостей**

**Модули — это не просто классы, а полноценные контейнеры с границами видимости:**

```
UsersModule {
  providers: [UsersService],     // Только внутри модуля
  exports: [UsersService],       // Доступно импортерам
  imports: [AuthModule]          // Доступ к AuthService (если exported)
}
```

**Граф зависимостей** (внутренне):

```
AppModule ──→ UsersModule ──→ AuthModule
  │             │                │
(AppService) (UsersService)  (AuthService)
```

**Cyclic Dependencies** решаются через `@ForwardRef()`:

```typescript
// Создает Proxy-объект, который разрешается после полной инициализации графа
@Module({
  imports: [forwardRef(() => AuthModule)], // Proxy
})
export class UsersModule {}
```

### **🌐 4. HTTP Request Pipeline — полный цикл обработки**

**Пошагово, что происходит с запросом `GET /users/1`:**

```
1. Express/Fastify → находит роут /users/:id
2. Global Guards (app.useGlobalGuards())
3. Controller Guards (@UseGuards на классе)
4. Method Guards (@UseGuards на методе)
5. Global Pipes (app.useGlobalPipes(ValidationPipe))
6. Method Pipes (@UsePipes(ValidationPipe))
7. Interceptors:before (@UseInterceptors())
8. Controller.method() → UsersService.findOne(id)
9. Interceptors:after
10. Response (JSON автоматически)
```

**Метаданные определяют порядок**:

```typescript
@Get('users/:id')
@UseGuards(AuthGuard, RolesGuard)  // Список guards из Reflect.metadata
@UsePipes(ValidationPipe)          // Список pipes
async findOne(@Param('id') id: number) {} // paramtypes из метаданных
```

### **🔧 5. Metadata Reflection System — магия декораторов**

**`reflect-metadata` + TypeScript** генерируют **скрытые метаданные**:

```typescript
// В runtime доступно:
Reflect.getMetadata("design:paramtypes", UsersController);
// → [UsersService] — типы параметров конструктора

Reflect.getMetadata("path", UsersController.findOne);
// → '/users/:id' — от @Get()

Reflect.getMetadata("guards", UsersController.findOne);
// → [AuthGuard, RolesGuard] — от @UseGuards()
```

**tsconfig.json обязательно**:

```json
{
  "compilerOptions": {
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true // Без этого ничего не работает!
  }
}
```

### **⚙️ 6. ExecutionContext — универсальный адаптер**

**Один интерфейс для всех транспортов** (HTTP, WebSocket, gRPC, GraphQL):

```typescript
interface ExecutionContext {
  getType(): "http" | "ws" | "rpc" | "graphql";
  getClass(): any; // Controller class
  getHandler(): any; // Method
  switchToHttp(): HttpArgumentsHost; // req, res
  switchToWs(): WsArgumentsHost; // ws client
}
```

**Guard универсальный**:

```typescript
@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    switch (context.getType<"http">()) {
      case "http":
        const req = context.switchToHttp().getRequest();
        return validateToken(req.headers.authorization);
      case "ws":
        const client = context.switchToWs().getClient();
        return validateWsToken(client.handshake);
    }
  }
}
```

### **📦 7. Platform Adapters — Express/Fastify под капотом**

**NestJS НЕ привязан к Express** — использует **абстрактный адаптер**:

```
NestJS Core → INestApplication → PlatformFactory → ExpressAdapter/FastifyAdapter
```

**ExpressAdapter** преобразует:

```typescript
// Внутри platform-express.adapter.ts
public createHandler(): RequestHandler {
  return (req: Request, res: Response, next: NextFunction) => {
    const context = this.createExecutionContext(req, res, next);
    return this.reqHandler(context); // NestJS pipeline
  };
}
```

**FastifyAdapter** (+30-50% производительности):

```bash
npm i @nestjs/platform-fastify fastify
```

### **🔍 8. Dependency Resolution — алгоритм поиска**

**Алгоритм поиска провайдера**:

```
1. Token = PrismaService (class)
2. Проверить ModuleRef.instances.has(PrismaService)
3. Проверить ModuleRef.providers (локальные)
4. Проверить imports → рекурсивно
5. Проверить global modules
6. Factory provider → useFactory()
```

**Circular Dependencies**:

```
UsersService → AuthService → UsersService (цикл!)
@ForwardRef(() => AuthModule) → Proxy(AuthService) → решается после init
```

### **🔥 Собеседование Middle+ вопросы**

**Q: "Как NestJS знает зависимости конструктора?"**

```
A: Reflect.getMetadata('design:paramtypes', Service) из emitDecoratorMetadata
```

**Q: "Что делает @ForwardRef()?"**

```
A: Создает Proxy для разрешения циклических зависимостей в Module DAG
```

**Q: "Порядок выполнения Guards/Pipes/Interceptors?"**

```
A: Global→Controller→Method (Guards→Pipes→Interceptors:before→Controller→after)
```

**Q: "Почему Fastify быстрее Express?"**

```
A: Native async/await, JSON parser, без callback hell
```

**Формула**: **IoC(Reflect) + ModuleDAG + ExecutionContext + PlatformAdapter** 🚀
