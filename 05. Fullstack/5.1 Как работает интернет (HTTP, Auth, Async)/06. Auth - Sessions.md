## 🔐 Sessions vs JWT Tokens: Полное сравнение

Sessions и JWT — два подхода к **stateful** и **stateless** авторизации. Разберём принципы работы, плюсы/минусы и когда использовать.

### **📊 Сравнительная таблица**

| Характеристика             | **Sessions (Stateful)**           | **JWT Tokens (Stateless)**                 |
| :------------------------- | :-------------------------------- | :----------------------------------------- |
| **Где хранится состояние** | Сервер (Redis/Memory/DB)          | Клиент (токен содержит все данные)         |
| **Идентификатор**          | Session ID (`abc123`)             | Полные данные (`header.payload.signature`) |
| **Размер**                 | Маленький ID (32 байта)           | Большой токен (500-2000 байт)              |
| **Масштабирование**        | Требует sticky sessions или Redis | ✅ Работает везде                          |
| **Logout**                 | ✅ Мгновенно (DELETE сессия)      | ❌ Только истечение срока                  |
| **Производительность**     | Медленнее (поиск в Redis)         | ✅ Быстрее (проверка подписи)              |

**Redis (Remote Dictionary Server)** — это очень быстрая, нереляционная (NoSQL) система управления базами данных с открытым исходным кодом, работающая по принципу «ключ-значение» и хранящая данные в оперативной памяти (RAM). Из-за хранения в RAM Redis обеспечивает субмиллисекундное время отклика и используется для кэширования, брокеров сообщений и хранения сессий.

### **🔄 Как работают**

#### **1. Sessions (традиционный подход)**

```
1. POST /login → email/password
2. Сервер: ✅ пароль верный
3. Сервер: Создаёт сессию {id: "abc123", userId: 456, exp: +24h}
4. ← Set-Cookie: sessionId=abc123; HttpOnly
5. GET /profile → Cookie: sessionId=abc123
6. Сервер: Redis.get("abc123") → {userId: 456} → 200 OK
```

**Express код**:

```javascript
app.post("/login", (req, res) => {
  const sessionId = generateId();
  req.session = { userId: 123, expires: Date.now() + 24 * 60 * 60 * 1000 };
  res.json({ message: "ok" });
});

app.get("/profile", (req, res) => {
  if (req.session?.userId) {
    res.json({ user: req.session.userId });
  } else {
    res.status(401).send();
  }
});
```

#### **2. JWT (ранее разбирали)**

```
1. POST /login → JWT токен с данными
2. GET /profile → Authorization: Bearer <jwt>
3. Сервер: jwt.verify(token) → {userId: 456} → 200 OK
```

### **⚖️ Плюсы и минусы**

| **Sessions**                                    | **JWT**                         |
| :---------------------------------------------- | :------------------------------ |
| ✅ **Мгновенный logout** (`req.session = null`) | ❌ Нельзя отозвать до истечения |
| ✅ **Сервер полный контроль**                   | ✅ **Масштабируется без Redis** |
| ✅ **Маленький размер** cookie                  | ❌ **Большой токен**            |
| ❌ **Redis кластер** нужен                      | ✅ **Stateless**                |
| ✅ **Простая инвалидация**                      | ❌ **Refresh token ротация**    |

### **🛡️ Безопасность**

| Угроза            | **Sessions**                       | **JWT**                         |
| :---------------- | :--------------------------------- | :------------------------------ |
| **XSS**           | `document.cookie` крадёт sessionId | `localStorage.getItem('token')` |
| **CSRF**          | SameSite=Strict                    | ✅ Только Bearer header         |
| **Token кража**   | ❌ Session hijacking               | ✅ Короткий TTL (15 мин)        |
| **Server breach** | Удалить все сессии                 | ✅ Подпись не поможет           |

### **🎯 Когда что использовать**

```
🟢 SESSIONS (традиционные сайты):
- Банки, e-commerce
- Много logout сценариев
- Один домен
- Redis уже есть

🟢 JWT (API/SPA/микросервисы):
- Мобильные приложения
- Микросервисы
- Много доменов
- Нет Redis
```

### **🚨 Типичные собес-запросы**

**Q: "Почему JWT лучше сессий?"**

```
❌ НЕТ универсального ответа!
✅ JWT: масштабируемость, микросервисы
✅ Sessions: контроль, logout, безопасность
```

**Q: "JWT stateless = безопасно?"**

```
❌ Stateless ≠ безопасно!
JWT украден = доступ 15 минут
Session ID украден = мгновенный logout
```

**Q: "Можно ли logout в JWT?"**

```
✅ Refresh token ротация (версия++)
✅ Blacklist expired tokens (Redis)
✅ Короткий TTL (5-15 мин)
```

### **💻 Гибридный подход (Production 2026)**

```
1. Access JWT: 15 минут (Bearer)
2. Refresh JWT: 7 дней (HttpOnly cookie)
3. Logout → blacklist refresh в Redis
```

**Итог Senior ответа**: **Sessions = контроль, JWT = масштабируемость. Выбор по архитектуре проекта** 🚀
