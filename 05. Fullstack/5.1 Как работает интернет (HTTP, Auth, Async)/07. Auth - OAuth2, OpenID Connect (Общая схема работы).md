## 🔐 OAuth2 + OpenID Connect: Подробное объяснение для понимания

OAuth2 и OpenID Connect кажутся сложными, но это просто **делегированная авторизация**. Разберём **шаг за шагом**, как это работает на практике.

### **🎭 Участники (роли) — представим реальный сценарий**

```
📱 Ты (Никита) хочешь зайти в блог-платформу "Дзен 2.0"
🔑 У Дзена нет своей системы логина/регистрации
🤝 Дзен говорит: "Залогинься через Яндекс ID — безопасно и быстро"
```

| Роль                     | Кто это                   | Пример                                  |
| :----------------------- | :------------------------ | :-------------------------------------- |
| **Resource Owner**       | **Ты (пользователь)**     | Никита Голов, владелец аккаунта         |
| **Client**               | **Дзен 2.0 (приложение)** | Твой React SPA или мобильное приложение |
| **Authorization Server** | **Яндекс ID**             | Сервис, который знает твои пароли       |
| **Resource Server**      | **API Дзена**             | `/api/posts`, `/api/profile`            |

### **🔄 Authorization Code Flow — пошагово (с примерами)**

#### **Шаг 1: Пользователь жмёт "Войти через Яндекс"**

```
Ты на Дзене → Кнопка "Войти через Яндекс ID"
↓
Дзен формирует специальную ссылку:
https://oauth.yandex.ru/authorize?
  client_id=dzrn0000000001&           ← ID приложения Дзен
  redirect_uri=https://dzrn.ru/callback& ← Куда вернуть
  response_type=code&                   ← Хотим код авторизации
  scope=openid email profile&           ← Какие данные нужны
  state=abcXYZ123                      ← Защита от CSRF
```

#### **Шаг 2: Яндекс показывает форму согласия**

```
Браузер переходит на Яндекс:
"Дзен 2.0 хочет:
✅ Читать твой email
✅ Видеть профиль
✅ Твой Яндекс ID

[Разрешить] [Запретить]"
```

#### **Шаг 3: Ты соглашаешься**

```
Ты жмёшь "Разрешить"
↓
Яндекс знает: Никита = user123@yandex.ru
Яндекс генерирует временный код: code=4efg567hij89
↓
Перенаправляет обратно:
https://dzrn.ru/callback?code=4efg567hij89&state=abcXYZ123
```

#### **Шаг 4: Дзен меняет код на токены (через Backend!)**

```
React SPA видит code → отправляет на Backend Дзена:
POST /api/auth/yandex/callback
{
  "code": "4efg567hij89"
}

Backend Дзена → Яндекс (секретно!):
POST https://oauth.yandex.ru/token
{
  "code": "4efg567hij89",
  "client_id": "dzrn0000000001",
  "client_secret": "СуперСекретныйПарольДзена",
  "grant_type": "authorization_code"
}
```

#### **Шаг 5: Яндекс выдаёт токены**

```json
{
  "access_token": "ya29.a0AfH6SMD...", // Для API Яндекса
  "id_token": "eyJhbGciOiJSUzI1NiIs...", // Кто ты такой
  "refresh_token": "1//04abc...", // Для обновления
  "expires_in": 3600
}
```

#### **Шаг 6: Backend Дзена создаёт свою сессию**

```
Backend Дзена:
1. decode(id_token) → {email: "nikita@test.ru", sub: "123456"}
2. Создаёт JWT Дзена или сессию
3. Set-Cookie: session_dzen=xyz789 (HttpOnly)
4. Redirect: /
```

### **🧩 Разница токенов (подробно)**

| Токен               | Содержит              | Используется для            | Пример содержимого                                   |
| :------------------ | :-------------------- | :-------------------------- | :--------------------------------------------------- |
| **`access_token`**  | Права доступа         | **API запросы**             | `GET /api/yandex/photos`                             |
| **`id_token`**      | **Информацию о тебе** | **Аутентификация**          | `{sub:"123", email:"nikita@test.ru", name:"Никита"}` |
| **`refresh_token`** | Ссылку на сессию      | **Обновление** access_token | Длинный срок жизни                                   |

**Почему Backend Proxy?**

```
❌ ПЛОХО: React SPA шлёт client_secret → любой увидит в Network
✅ ХОРОШО: Backend Дзена имеет client_secret → безопасно
```

### **📐 Полная диаграмма (текстовая)**

```
[1] Никита → Дзен: "Войти через Яндекс"
        ↓ GET /auth/yandex
[2] Дзен → Яндекс: authorize?client_id=dzrn...
        ↓ 302 Redirect
[3] Яндекс → Никита: "Разрешить доступ Дзену?"
        ↓ Никита: "Да"
[4] Яндекс → Дзен: /callback?code=abc123
        ↓ React SPA видит code
[5] React → Backend Дзена: POST /exchange {code: "abc123"}
        ↓ Backend → Яндекс /token (client_secret!)
[6] Яндекс → Backend: {access_token, id_token, refresh_token}
        ↓ Backend создаёт свою сессию/JWT Дзена
[7] Backend → React: Set-Cookie + {user: {name: "Никита"}}
        ↓
[8] Никита → Дзен: GET /profile (Cookie автоматом)
        ↓ Дзен: "Всё ок, это Никита"
```

### **🚨 Почему это безопасно?**

1. **Пароль Яндекса НИКОГДА не уходит к Дзену**
2. **client_secret только на Backend** (не в браузере)
3. **Короткие access_token** (1 час)
4. **state параметр** защищает от CSRF

### **💻 Реальный код (упрощённый)**

**React (`LoginButton`)**:

```typescript
const yandexLogin = () => {
  const state = crypto.randomUUID();
  localStorage.setItem("oauth_state", state); // CSRF защита

  const url =
    `https://oauth.yandex.ru/authorize?` +
    new URLSearchParams({
      client_id: "dzrn0000000001",
      redirect_uri: "https://dzrn.ru/api/auth/yandex/callback",
      response_type: "code",
      scope: "openid email profile",
      state,
    });

  window.location.href = url;
};
```

**Express Backend (`/api/auth/yandex/callback`)**:

```javascript
app.get("/api/auth/yandex/callback", async (req, res) => {
  const { code, state } = req.query;

  // Проверяем state (CSRF)
  if (state !== req.cookies.oauth_state) {
    return res.status(400).json({ error: "Invalid state" });
  }

  // Меняем code на токены
  const tokens = await exchangeCodeForTokens(code);
  const userInfo = await decodeIdToken(tokens.id_token);

  // Создаём свою сессию Дзена
  req.session.user = { id: userInfo.sub, email: userInfo.email };

  res.redirect("/dashboard");
});

// Получение токенов от Yandex
const exchangeCodeForTokens = async (code) => {
  return axios
    .post("https://oauth.yandex.ru/token", null, {
      params: {
        code,
        client_id: process.env.YANDEX_CLIENT_ID,
        client_secret: process.env.YANDEX_CLIENT_SECRET,
        grant_type: "authorization_code",
      },
    })
    .then((res) => res.data); // {access_token, id_token, refresh_token}
};
```

### **🎯 Ключевые выводы для собеседования**

```
1. OAuth2 = "ДАЙ ДОСТУП" к ресурсам (фото, почта)
2. OpenID Connect = "КТО ТЫ" (имя, email)
3. Authorization Code + Backend Proxy = безопасно для SPA
4. Никогда не храни client_secret в браузере!
```

**Простыми словами**: Ты даёшь Дзену доверенность на Яндекс, Дзен показывает доверенность Яндексу, Яндекс выдаёт Дзену временный пропуск для тебя. 🚀

## 🔐 PKCE и nonce — подробное объяснение

### **🧩 PKCE (Proof Key for Code Exchange)**

**Проблема**: SPA/мобильные приложения **не могут** хранить `client_secret` (видно в Network). Злоумышленник может перехватить `code` и обменять на токены.

**PKCE решает**: "Только **я** могу обменять этот code на токены".

#### **🔄 Как работает PKCE (4 шага)**

```
1. SPA генерирует секретный code_verifier (43-128 символов)
   code_verifier = "E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM"

2. SPA создаёт code_challenge = SHA256(code_verifier) → Base64
   code_challenge = "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"

3. SPA шлёт в /authorize:
   https://auth.yandex.ru/authorize?
     code_challenge=dBjftJeZ4CVP...&
     code_challenge_method=S256

4. При обмене code → tokens:
   SPA шлёт code_verifier
   Yandex: SHA256(verifier) === challenge? ✅ → токены
```

#### **💻 Реальный код React**

```javascript
// Генерация PKCE
const generatePKCE = () => {
  const verifier = generateRandomString(43); // Секрет
  const challenge = base64URLEncode(sha256(verifier)); // Публичный

  return { code_verifier: verifier, code_challenge: challenge };
};

// В URL авторизации
const { code_verifier, code_challenge } = generatePKCE();
const params = new URLSearchParams({
  client_id: "dzrn123",
  code_challenge,
  code_challenge_method: "S256", // SHA256
  // ... другие параметры
});
```

**Если злоумышленник перехватит `code`**: **не знает `code_verifier`** → не получит токены!

### **🎲 nonce (Number used ONCE)**

**Проблема**: Злоумышленник перехватил `id_token` и пытается **повторно использовать** (replay attack).

**nonce решает**: "Этот `id_token` **свежий**, не переиспользован".

#### **🔄 Как работает nonce**

```
1. SPA генерирует nonce = crypto.randomUUID()
   nonce = "8f782ac2-f68d-11ee-be56-0242ac120002"

2. SPA шлёт в /authorize:
   nonce=8f782ac2-f68d-11ee-be56-0242ac120002

3. Yandex кладёт nonce В id_token:
   id_token = {
     sub: "user123",
     nonce: "8f782ac2-f68d...",  // ← копия!
     ...
   }

4. SPA проверяет при получении:
   jwt.decode(id_token).nonce === мой_nonce ? ✅
```

#### **💻 Проверка nonce**

```javascript
const expectedNonce = localStorage.getItem("oauth_nonce");
const idToken = tokens.id_token;
const decoded = jwt.decode(idToken);

if (decoded.nonce !== expectedNonce) {
  throw new Error("Replay attack detected!");
}
```

### **⚡ Сравнение PKCE vs nonce**

| **PKCE**                           | **nonce**                        |
| :--------------------------------- | :------------------------------- |
| Защищает **code → tokens** обмен   | Защищает **id_token** от replay  |
| `code_verifier/challenge`          | `random string → id_token.nonce` |
| **Обязателен** для SPA (OAuth 2.1) | **Рекомендуется** для OIDC       |
| SHA256 хэш                         | Простая строка                   |

### **🚨 Типичные собес-запросы**

**Q: "Зачем PKCE если есть HTTPS?"**

```
HTTPS защищает от MITM (перехват)
PKCE защищает от злоумышленника, который:
1. Увидел code в redirect_uri
2. Знает client_id (публичный)
❌ Без PKCE → обмен code на токены
✅ С PKCE → нужен code_verifier
```

**Q: "PKCE только для публичных клиентов?"**

```
✅ OAuth 2.1: PKCE **ОБЯЗАТЕЛЕН** для ВСЕХ клиентов в Code Flow
Раньше: только public clients (SPA/mobile)
```

### **🎯 Ключевые формулы**

```
PKCE: code_challenge = SHA256(code_verifier)
Проверка: SHA256(verifier) === challenge?

nonce: random_string → id_token.nonce
Проверка: id_token.nonce === expected_nonce?
```

**Запомни**: **PKCE = защита code обмена, nonce = защита id_token**
