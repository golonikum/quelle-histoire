# HTTP/HTTPS: Основные методы (CRUD)

**HTTP** — протокол передачи данных, **HTTPS** — HTTP + шифрование TLS/SSL. Методы определяют **действие** с ресурсом.

## 📊 Сравнение методов

| Метод      | Назначение           | Безопасный¹ | Идемпотентный² | Тело запроса | Кэшируется | REST CRUD  |
| :--------- | :------------------- | :---------- | :------------- | :----------- | :--------- | :--------- |
| **GET**    | Получить данные      | ✅ Да       | ✅ Да          | ❌ Нет       | ✅ Да      | **READ**   |
| **POST**   | Создать ресурс       | ❌ Нет      | ❌ Нет         | ✅ Да        | ❌ Нет     | **CREATE** |
| **PUT**    | Заменить целиком     | ❌ Нет      | ✅ Да          | ✅ Да        | ❌ Нет     | **UPDATE** |
| **PATCH**  | Частичное обновление | ❌ Нет      | ❌ Нет³        | ✅ Да        | ❌ Нет     | **UPDATE** |
| **DELETE** | Удалить ресурс       | ❌ Нет      | ✅ Да          | ❌/✅        | ❌ Нет     | **DELETE** |

**¹ Безопасный** = не меняет состояние сервера
**² Идемпотентный** = N одинаковых вызовов = 1 результат
**³ PATCH** может быть идемпотентным в зависимости от реализации

## 🔍 Подробно по каждому

### **GET** — "Дай мне данные"

```
GET /api/users/123
→ 200 OK + JSON { id: 123, name: "Иван" }
```

- Параметры в URL: `/users?age=25&city=Москва`
- **Кэшируется** браузером
- **Повторный вызов** = тот же результат

### **POST** — "Создай новый"

```
POST /api/users
Content-Type: application/json
{ "name": "Петр", "email": "petr@test.ru" }
→ 201 Created + { id: 456, name: "Петр" }
```

- **НЕ идемпотентен**: 2 POST = 2 пользователя
- Тело запроса обязательно

### **PUT** — "Замени целиком"

```
PUT /api/users/123
{ "name": "Иван Updated", "email": "ivan2@test.ru", "age": 31 }
→ 200 OK + полный объект
```

- **Идемпотентен**: 2 одинаковых PUT = 1 результат
- **Все поля обязательны** (или null)

### **PATCH** — "Измени частично"

```
PATCH /api/users/123
{ "email": "ivan3@test.ru" }  // Только email!
→ 200 OK + { id: 123, name: "Иван Updated", email: "ivan3@test.ru", age: 31 }
```

- Остальные поля **не трогаются**
- Экономит трафик

### **DELETE** — "Удали"

```
DELETE /api/users/456
→ 204 No Content (или 200 OK)
```

- **Идемпотентен**: DELETE дважды = ресурс удалён

## 💻 Примеры fetch (JavaScript)

```javascript
// GET
fetch("/api/users/123")
  .then((res) => res.json())
  .then((user) => console.log(user));

// POST
fetch("/api/users", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "Петр", email: "petr@test.ru" }),
});

// PUT
fetch("/api/users/123", {
  method: "PUT",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    name: "Иван Updated",
    email: "ivan2@test.ru",
    age: 31,
  }),
});

// PATCH
fetch("/api/users/123", {
  method: "PATCH",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ email: "ivan3@test.ru" }),
});

// DELETE
fetch("/api/users/456", { method: "DELETE" });
```

## 🎯 RESTful API соответствие

```
GET    /users           → список пользователей (READ)
GET    /users/123       → один пользователь (READ)
POST   /users           → создать (CREATE)
PUT    /users/123       → заменить (UPDATE)
PATCH  /users/123       → частично (UPDATE)
DELETE /users/123       → удалить (DELETE)
```

## 🚨 Типичные ошибки

```
❌ GET с телом запроса (запрещено стандартом)
❌ POST для чтения данных
❌ PUT без полного объекта
✅ POST для создания, PUT/PATCH для обновления
```

## 🔐 HTTPS vs HTTP

```
HTTP  → незашифровано (пароли в Network!)
HTTPS → TLS/SSL шифрование + HSTS
```

**TL;DR**: **GET=читать, POST=создать, PUT=заменить полностью, PATCH=починить частично, DELETE=удалить**. Идемпотентность критично для retry логики! 🚀

## 🔥 Вопросы на собеседовании по HTTP методам (Middle/Senior)

### **1. Базовые (Middle)**

**Q: В чём разница GET и POST?**

```
GET: читать данные, параметры в URL, кэшируется, идемпотентен
POST: создавать данные, тело запроса, НЕ кэшируется, НЕ идемпотентен
```

**Q: Какие HTTP методы вы знаете?**
**Ответ**: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS, TRACE.

### **2. Идемпотентность (Middle+)**

**Q: Какие методы идемпотентны? Приведите примеры.**

```
✅ Идемпотентны: GET, PUT, DELETE, HEAD, OPTIONS
✅ НЕ идемпотентны: POST, PATCH

PUT /users/123 дважды = 1 результат ✅
POST /users дважды = 2 пользователя ❌
```

**Q: Что будет если повторить DELETE /users/123 дважды?**
**Ответ**: 200/204 оба раза — ресурс уже удалён (идемпотентность).

### **3. PUT vs PATCH (Middle+)**

**Q: PUT или PATCH для смены email пользователя?**

```
PUT /users/123 → весь объект {name, email, age}
PATCH /users/123 → {email: "new@test.ru"}
```

**Ответ**: PATCH (экономия трафика).

**Q: Можно ли использовать POST для чтения данных?**
**Ответ**: Технически да, но **REST-антипаттерн**. GET для чтения.

### **4. Live-coding задачи (Senior)**

**Q: Напишите CRUD API endpoints для пользователей:**

```
✅ GET /api/users           → список (200)
✅ GET /api/users/123       → один (200/404)
✅ POST /api/users          → создать (201)
✅ PUT /api/users/123       → заменить (200/404)
✅ PATCH /api/users/123     → частично (200/404)
✅ DELETE /api/users/123    → удалить (204/404)
```

**Q: Типизировать fetch запросы в TS:**

```typescript
interface User {
  id: number;
  name: string;
  email: string;
}

async function getUser(id: number): Promise<User> {
  return fetch(`/api/users/${id}`).then((res) => {
    if (!res.ok) throw new Error("404");
    return res.json();
  });
}
```

### **5. Безопасность + Headers (Senior)**

**Q: Почему GET с паролем в query string — плохо?**

```
❌ GET /login?password=123 → лог браузера, реферер, кэш!
✅ POST /login → тело запроса (не логируется)
```

**Q: Content-Type заголовки?**

```
application/json → API
multipart/form-data → файлы
application/x-www-form-urlencoded → формы
```

### **6. Статус-коды (Middle/Senior)**

**Q: Разница 401 vs 403?**

```
401 Unauthorized → нет токена/пароля
403 Forbidden → токен есть, прав нет
```

**Q: Когда возвращать 200 vs 201?**

```
200 OK → данные получены/обновлены
201 Created → новый ресурс создан (POST)
204 No Content → DELETE успешно
```

### **7. Продвинутые подставы (Senior)**

**Q: Можно ли GET с телом запроса?**
**Ответ**: **Можно**, но **стандарт запрещает**. Серверы игнорируют.

**Q: POST идемпотентен?**
**Ответ**: **Нет**. 2 POST = 2 ресурса. Исключение: API с `Idempotency-Key`.

**Q: HEAD vs GET?**

```
HEAD: только headers (проверка существования)
GET: headers + тело
```
